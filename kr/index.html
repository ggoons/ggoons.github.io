소개

꾼즈는 코딩 대행 서비스 입니다. 사용자께서 개발을 원하는 목표 시스템의 화면/페이지 명세서 를 꾼즈에 보내 주시면 꾼즈는 이 명세서를 구현 한 백엔드, 프론트앤드 소스코드를 다운로드할 수 있도록 제공 합니다. 꾼즈에 의해 제공되는 소스코드가 아직 최고의 품질은 아니지만 필요한 사용자들이 있을수도 있다고 판단하여 서비스를 시작 합니다.

이용대상

꾼즈는 사용자의 요청에 의해 데이터 입출력을 다루는 소스 코드를 생산 합니다. 꾼즈가 사용자에게 제공하는 소스 코드는 사용자가 작성한 명세서의 기능 구현에만 집중 할 뿐, 세련된 UI 를 목표로 하는 최종 사용자가 사용할 수 있는 완성된 시스템이 아닙니다. 따라서 꾼즈 서비스의 이용고객은 꾼즈가 제공하는 소스코드를 유지 관리 할 수 있는 개발자들 입니다.

이용절차

  • 사용자가 코드 생성 요청서를 시스템 명세서와 함께 ggoon@ggoons.com 으로 보내시면 코드 생성 계약이 시작 됩니다.
  • 이메일의 첫 문장으로 "꾼즈 홈 페이지의 이용방법및 가격 정책을 읽었고 충분히 이해 했으며, 코드생성을 요청 합니다." 로 작성 하십시요.
    이 문장으로 시작하지 않는 요청서는 무시 됩니다.
  • 그 다음 백엔드 종류 및 프론트엔드 종류를 지정 하십시요. (각 디바이스 별 하나 씩 가능.)
  • 백엔드의 DBMS 종류를 지정 하십시요.
  • 회사명과 프로젝트명을 지정 하십시요. (있는 경우에만) 이 이름들은 소스코드의 패키지명에 사용 됩니다.
  • 시스템 명세서는 zip 파일로 압축한 후 첨부 파일로 추가해 주십시요.
  • 만약 명세서를 손 글씨로 작성 하였다면 사진을 찍어서 zip 파일에 추가 하십시요.
  • 샘플: 
    꾼즈 홈 페이지의 이용방법및 가격 정책을 읽었고 충분히 이해 했으며, 코드생성을 요청 합니다.

첨부파일: xxx

  • 꾼즈에서 명세서를 분석한 후, 가격을 통보 합니다.
  • 사용자는 꾼즈에서 통보한 가격의 10% 를 계약금으로 입금 해야 합니다.
  • 꾼즈에서 계약금 입금이 확인되면, 코드를 생성한 후, 실행 가능 주소를 사용자에게 통보 합니다.
  • 사용자는 완성된 실행 화면을 검토한 후, 구매 여부를 결정 합니다.
  • 생성된 결과가 만족스럽지 못하다면 구매를 거부할 수 있습니다. 이 경우, 계약금은 반환하지 않습니다.
  • 만약 구매를 결정 한다면, 나머지 잔금을 입금 합니다.
  • 꾼즈는 잔금 입금이 확인되면 사용자에게 전체 소스 코드를 전송 하고, 계약은 종료 합니다.

  • 일단 거래가 종료된 후, 기본적으로 꾼즈 에서는 관련 자료를 모두 폐기 합니다.
  • 그러나 사용자가 이 후에 테이블 이나 컬럼 의 추가 삭제를 원한다면 수정 대상으로 지정 할 수 있습니다.
  • 프로젝트가 수정대상이 되면, 명세서는 유지 되며, 이 후의 수정 요청은 추가 삭제된 페이지의 컬럼 수 만큼 만 요금이 발생 합니다.
  • 수정대상이 아닌 경우, 명세서및 생성된 모든 코드는 꾼즈 서버에서 폐기 되며 이 후의 수정 요청은 추가 제출 된 명세서의 모든 컬럼 수 만큼 요금이 발생 합니다.

  • 여기서 언급하는 수정은 명세서의 수정을 의미 합니다.
  • 즉, 생성된 소스코드의 수정을 의미 하는 것이 아닙니다. 소스코드는 명세서를 기반으로 항상 새롭게 생성 됩니다.
  • 따라서 일단 생성된 소스코드를 사용자가 수 작업으로 수정 하던 중 명세서를 수정한 경우라면 새로운 명세서에 맞게 소스코드를 직접 수정 하셔야 합니다.

명세서 작성시 참고

명세서에는 정해진 형식이 없습니다. 자유롭게, 개발을 원하는 시스템에 속한 각각의 화면(이하 페이지 라 함) 에 어떤 데이터 들이 입/출력 되어야 하는지 기능을 정의 하십시요. 다음은 몇가지 주의 사항 입니다.

  • 명세서는 페이지 단위로 작성 하십시요.
  • 페이지에 어떤 데이터가 입출력 되어야 하는지 명시 하십시요.
  • 데이터 입출력 기능 명세 이외에는 무시 됩니다. (예: 버튼 색상, 폰트 크기, 등등)
  • 외부 시스템은 apikey 또는 OAuth 2.0 으로 접근 가능한 경우에만 지원 합니다.
  • 소셜 로그인 이나 외부 시스템 데이터를 사용하는 경우, 해당 외부 시스템에 대한 스키마를 제공 해야 합니다.
    여기서 스키마란, 외부 시스템에 접근하기 위한 주소및 외부 시스템의 요청/응답 데이터 구조를 의미 합니다.

  • 데이터베이스 스키마가 제공된 경우:
    • 스키마는 MySQL 용 SQL 파일로 제공 되어야 합니다.
    • 스키마는 명세서를 구현 하는데 충분한 무결성을 갖고 있어야 합니다.
    • 페이지 기능 정의 시, 테이블 이름과 컬럼 이름을 사용 하십시요.
    • 페이지에 지도가 표시되어야 하는 경우, 경도/위도 또는 주소 정보가 테이블에 존재 해야 합니다.
    • 상태 관리가 필요한 테이블이 있다면 해당 테이블명을 상태관리 테이블 목록에 추가 하십시요.
      ex) 상태관리 테이블: 테이블1, 테이블2, ...
    • 만약 상태관리 테이블이 존재 한다면, 상태관리 라이브러리를 선택 하십시요. ( ngrx 또는 ngxs )
    • 페이지에 사용된 데이터의 출처가 분명 해야 합니다. (테이블 또는 외부 시스템)

  • 통계 및 차트는 사용되는 DB 가 RDB 인 경우에만 허용 됩니다.
  • 소스코드의 TAB 을 TAB 으로 할 지, 아니면 n 개의 공백으로 할 지 지정 하십시요.
  • 버튼이나 배치작업등 특정 이벤트 발생 시, 실행할 CRUD 로직이 있으면, 해당 로직을 pseudo code 로 작성 하십시요.

가격

가격은 프로젝트 단위와 컬럼 단위로 나누어 집니다.

프로젝트 단위

이 상품은 고객과 꾼즈가 직접 대면하여 계약서를 작성한 후 진행 합니다. 프로젝트 단위 가격은 프로젝트 비용의 22%(부가세 포함) 입니다. 이 경우 프로젝트의 시작 부터 설계가 완료될 때 까지 설계의 구현을 무한정 제공 합니다. 또한 이 후 UI 개선 개발자(들) 에게 현재까지 생성된 소스코드의 상세 설명을 제공 합니다.

컬럼 단위

가격은 생성되는 소스코드의 종류(백엔드, 프론트엔드)나 수량에 무관하게 명세서의 모든 페이지에 포함된 컬럼 수 * 컬럼 단가로 결정 되며 컬럼 단가는 다음과 같이 결정 됩니다.

데이터베이스 스키마 컬럼 단가 비고
미제공 22,000원 부가세 포함
제공 2,200원 부가세 포함

데이터베이스 스키마가 제공된 경우와 그렇지 않은 경우, 단가 차이가 많이 나므로 사용자의 주의가 필요 합니다. 또한 제공된 명세서나 데이터베이스 스키마에 오류가 있는 경우, 그 오류의 정도에 따라 계약이 파기 될 수도 있슴을 인지 하셔야 합니다. 작은 오류는 조정 되어 진행 되지만 너무 많은 오류가 있는 경우 프로젝트의 계속 진행이 불가 할 수도 있습니다. 그리고 이 경우, 사용자가 입금한 계약금은 반환하지 않습니다.

약관

사용자가 꾼즈에 코드 생성을 요청 하면 아래의 약관에 동의 한 것으로 간주 됩니다.

  • 사용자(이하 고객이라 함) 가 소스코드 생성을 의뢰하고 꾼즈(이하 회사라 함) 가 생산한 소스코드(이하 소스코드라 함)의 책임과 권한에 대해 정의 합니다.
  • 소스코드의 저작권은 고객과 회사가 공유 합니다.
  • 회사는 다른 고객의 의뢰에 의해 비슷한 소스코드를 생산할 권리가 있습니다.
  • 회사는 소스코드의 수정/배포 권한이 없습니다.
  • 고객은 소스코드의 수정/배포 권한이 있습니다.
  • 회사는 고객이 명세서에 명시한 데이터 입/출력 기능에 이상이 있으면 해당 기능의 정상화를 위해 무료로 수정 할 책임이 있습니다.
  • 고객은 고객이 명세서에 명시한 데이터 입/출력 기능에 이상이 있는지 여부를 검증할 책임이 있습니다.
  • 회사는 고객의 요청에 의해 고객에게 제공한 소스코드의 동작에 대해 어떠한 책임도 지지 않습니다.
